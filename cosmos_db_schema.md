# Azure Cosmos DB Schema Design for Virtual Queue System

This document outlines the proposed schema for the Azure Cosmos DB, designed to support the Virtual Queue Cafeteria Listing and Ordering System. The schema is structured to optimize for common access patterns, ensuring efficient data retrieval and scalability.

## 1. Database and Containers

We will use a single Azure Cosmos DB database with multiple containers, each representing a core entity in the system. The choice of partition key for each container is critical for performance and cost optimization.

## 2. Container Details and Schema

### 2.1. `users` Container

*   **Purpose:** Stores user profiles for both students and vendors.
*   **Partition Key:** `/id`
*   **Example Document:**

```json
{
  "id": "u12345",
  "name": "Arjun M",
  "email": "arjun@college.edu",
  "passwordHash": "hashed_password_string", // Store hashed passwords
  "role": "student", // "student" or "vendor"
  "rollNo": "20CS001", // For students
  "shopName": "",
  "shopType": "",
  "cafeId": null,
  "setupDone": false
}
```

### 2.2. `cafeterias` Container

*   **Purpose:** Stores information about each food outlet.
*   **Partition Key:** `/id`
*   **Example Document:**

```json
{
  "id": "cafe1",
  "name": "Main Canteen",
  "type": "Food",
  "emoji": "🍱",
  "open": true,
  "vendorId": "v12345", // ID of the vendor managing this cafeteria
  "description": "Main cafeteria — hot meals all day"
}
```

### 2.3. `menuItems` Container

*   **Purpose:** Stores the menu items offered by each cafeteria.
*   **Partition Key:** `/cafeId`
*   **Example Document:**

```json
{
  "id": "item101",
  "cafeId": "cafe1",
  "name": "Veg Thali",
  "desc": "Rice, dal, sabzi, roti, salad",
  "price": 60,
  "category": "Food",
  "emoji": "🍱",
  "available": true
}
```

### 2.4. `orders` Container

*   **Purpose:** Stores all placed orders, including details of items, student, and status.
*   **Partition Key:** `/cafeId` (allows vendors to efficiently query their orders)
*   **Example Document:**

```json
{
  "id": "ord12345",
  "cafeId": "cafe1",
  "cafeName": "Main Canteen",
  "studentId": "u12345",
  "studentName": "Arjun M",
  "items": [
    { "itemId": "item101", "name": "Veg Thali", "qty": 1, "price": 60 },
    { "itemId": "item201", "name": "Mango Shake", "qty": 1, "price": 45 }
  ],
  "total": 105,
  "token": 101,
  "status": "ordered", // "ordered", "preparing", "ready", "completed", "expired"
  "time": "10:30 AM",
  "date": "2026-04-09",
  "orderClaimed": false // For one-time claim protection
}
```

### 2.5. `queues` Container

*   **Purpose:** Manages the token system for each cafeteria.
*   **Partition Key:** `/cafeId`
*   **Example Document:**

```json
{
  "id": "queue_cafe1", // Unique ID for the queue document
  "cafeId": "cafe1",
  "lastToken": 100, // The last token issued for this cafeteria
  "currentServingToken": 100 // The token currently being served
}
```

## 3. Azure Functions Backend Design

Azure Functions will serve as the serverless backend, interacting with Cosmos DB to handle all data operations. Each function will expose an HTTP endpoint that the frontend can call.

### 3.1. Authentication and User Management

*   **`AuthLogin` (HTTP POST):**
    *   **Input:** `email`, `password`
    *   **Output:** User object (excluding password hash) or error.
    *   **Logic:** Authenticate user against `users` container. Generate a JWT token for authenticated sessions.

*   **`AuthRegister` (HTTP POST):**
    *   **Input:** `name`, `email`, `password`, `role`, `rollNo` (optional), `shopName` (optional), `shopType` (optional)
    *   **Output:** New user object (excluding password hash) or error.
    *   **Logic:** Create a new user in the `users` container. Hash the password before storing.

*   **`UpdateUserProfile` (HTTP PUT):**
    *   **Input:** User ID, updated user data.
    *   **Output:** Updated user object.
    *   **Logic:** Update user details in the `users` container.

### 3.2. Cafeteria and Menu Management

*   **`GetCafeterias` (HTTP GET):**
    *   **Input:** None
    *   **Output:** List of all cafeterias.
    *   **Logic:** Retrieve all documents from the `cafeterias` container.

*   **`GetCafeteriaById` (HTTP GET):**
    *   **Input:** `cafeId`
    *   **Output:** Single cafeteria object.
    *   **Logic:** Retrieve a specific cafeteria from the `cafeterias` container.

*   **`SetupShop` (HTTP POST):**
    *   **Input:** `vendorId`, `shopName`, `shopType`, `description`
    *   **Output:** New cafeteria object and updated vendor user object.
    *   **Logic:** Create a new cafeteria in the `cafeterias` container and update the vendor's `cafeId` and `setupDone` status in the `users` container.

*   **`GetMenuItems` (HTTP GET):**
    *   **Input:** `cafeId`
    *   **Output:** List of menu items for the specified cafeteria.
    *   **Logic:** Query `menuItems` container using `cafeId` as partition key.

*   **`AddMenuItem` (HTTP POST):**
    *   **Input:** `cafeId`, `name`, `desc`, `price`, `category`, `emoji`
    *   **Output:** New menu item object.
    *   **Logic:** Add a new item to the `menuItems` container.

*   **`UpdateMenuItemAvailability` (HTTP PUT):**
    *   **Input:** `cafeId`, `itemId`, `available`
    *   **Output:** Updated menu item object.
    *   **Logic:** Update the `available` status of a menu item in the `menuItems` container.

*   **`DeleteMenuItem` (HTTP DELETE):**
    *   **Input:** `cafeId`, `itemId`
    *   **Output:** Success confirmation.
    *   **Logic:** Delete a menu item from the `menuItems` container.

### 3.3. Order and Queue Management

*   **`PlaceOrder` (HTTP POST):**
    *   **Input:** `cafeId`, `studentId`, `items` (array of `itemId`, `qty`, `price`), `cafeName`, `studentName`
    *   **Output:** New order object with token.
    *   **Logic:** Increment `lastToken` in the `queues` container for the given `cafeId`, create a new order in the `orders` container with the new token.

*   **`GetOrdersByCafeId` (HTTP GET):**
    *   **Input:** `cafeId`
    *   **Output:** List of orders for the specified cafeteria.
    *   **Logic:** Query `orders` container using `cafeId` as partition key.

*   **`GetOrdersByStudentId` (HTTP GET):**
    *   **Input:** `studentId`
    *   **Output:** List of orders for the specified student.
    *   **Logic:** Query `orders` container using `studentId` (cross-partition query or secondary index).

*   **`UpdateOrderStatus` (HTTP PUT):**
    *   **Input:** `orderId`, `newStatus`
    *   **Output:** Updated order object.
    *   **Logic:** Update the `status` of an order in the `orders` container.

*   **`GetQueueStatus` (HTTP GET):**
    *   **Input:** `cafeId`
    *   **Output:** Current queue status (`lastToken`, `currentServingToken`).
    *   **Logic:** Retrieve the queue document for the specified `cafeId` from the `queues` container.

*   **`AdvanceQueue` (HTTP PUT):**
    *   **Input:** `cafeId`
    *   **Output:** Updated queue status.
    *   **Logic:** Increment `currentServingToken` in the `queues` container for the given `cafeId`.

*   **`VerifyOrderClaim` (HTTP POST):**
    *   **Input:** `orderId` (or QR code data containing `orderId`)
    *   **Output:** Order details if valid and not yet claimed, or error.
    *   **Logic:** Retrieve order, check `orderClaimed` status. If not claimed, mark as claimed and return order details. This is for the "Cross-Cafeteria Order Verification" and "One-Time Claim Protection" features.

### 3.4. Analytics and Notifications (Future Enhancements)

*   **`GetVendorAnalytics` (HTTP GET):**
    *   **Input:** `cafeId`, `timeRange`
    *   **Output:** Analytics data (peak hours, most ordered items, average prep time).
    *   **Logic:** Aggregate data from `orders` container. This might involve more complex Cosmos DB queries or integration with Azure Synapse Analytics.

*   **`OrderExpiryCheck` (Timer Triggered Function):**
    *   **Input:** None (triggered on a schedule)
    *   **Output:** Logs of expired orders.
    *   **Logic:** Scan `orders` container for orders past their expiry time and update their status to `expired`.

## 4. Connection String Configuration

The `COSMOS_CONNECTION_STRING` will be stored as an application setting in the Azure Static Web App and Azure Functions. This connection string will allow the Azure Functions to securely connect to the Cosmos DB instance.

## 5. Initial Data Insertion

As specified, initial data for `queues`, `cafeterias`, `users`, and `menuItems` will need to be inserted directly into Cosmos DB after container creation. This can be done via the Azure Portal Data Explorer or using a script.
